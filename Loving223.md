# 2025-08-17

Web3 新人速融职场｜真诚上线版

结论：别急着“显得专业”，先做到“可被验证的专业”。用小步快跑的交付，赢来下一次协作的门票。


0. 一句话心法

知道 ≠ 做过。少包装、多交付；少宣言、多链接（仓库、作品、截图、上线地址）。圈子小，口碑像链上交易——留痕且不可逆。


1. 入场三步走（7 天就能起量）
	•	第 1–2 天｜抄作业：跟一个你仰慕的项目，做一份“公开可见”的 Competitor/Community/Token 速研卡（1 页图 + 要点）。
	•	第 3–5 天｜小交付：挑一个真实痛点做微产品/流程优化：
	•	示例：把项目 Twitter/Discord 的 FAQ 做成一页 Notion/飞书；或把中文社媒评论做周报。
	•	第 6–7 天｜公开证明：把交付挂 GitHub/Notion，附可验证证据（链接、截图、数据）。连发 2 条 X（Twitter）复盘帖，主动 @ 项目方/核心成员。


2. Web3 vs. 传统职场（速览表）

维度	Web3	传统
组织	扁平+去中心，强 Owner 意识	科层清晰，岗位边界稳
协作	远程+异步为主，公开透明	同步会议多，信息分层
评价	以产出为王（链接可证）	以过程+汇报为主
节奏	市场驱动、波动大	规划驱动、节奏稳
风险	薪酬结构复杂、合规灰度	劳动法框架更完善

友情提示：并非“不涉及纳税”。不同地区对 U/代币收入多为应税所得，请自行合规（非税务建议）。

# 2025-08-15

“集中流动性”（Concentrated Liquidity）是 Uniswap v3 最核心的创新，它打破了之前 v2 那种“全价区无限铺水”的粗放做法，让 LP（流动性提供者）能像“狙击手”一样，把资金精准地放在目标价格区间，从而提高资金利用率，减少“闲着没事的资本”。

我给你拆开讲——

1️⃣ 传统 AMM（v2）的浪费问题
   •   在 v2 里，资金是从 0 到 ∞ 价格区间均匀分布的。
   •   如果 ETH 现在是 $2,000，你的资金大部分其实在远离当前价格的区间里“躺平”，几乎不会被交易触发。
   •   结果：资金利用率极低，LP 需要提供更多资金才能赚到同样的手续费。

2️⃣ v3 的集中流动性玩法
   •   定义：LP 可以指定一个价格区间（比如 $1,800 ~ $2,200）来提供流动性。
   •   在这个区间内，资金被“压缩”成更密集的流动性，交易撮合时更容易被用到。
   •   好处：
      •   资本效率提升（官方说法：最多可提升 4,000 倍）
      •   同等资金下，赚到的手续费更多
      •   LP 可以根据行情动态调整区间，做类似“被动做市 + 半主动交易”的混合玩法。

3️⃣ 举个直观例子

假设你有 10 ETH + 20,000 USDC：
   •   v2 模式：
      •   资金均匀铺在 0~∞ 的价格区间
      •   只有极小一部分流动性处在当前价格附近被用到
   •   v3 集中流动性模式（$1,900 ~ $2,100）：
      •   资金集中在这个窄区间
      •   大部分交易量在区间内发生
      •   你的手续费收入显著增加

4️⃣ 风险与代价
   •   如果价格跑出你设置的区间，你的流动性就会完全变成一种资产（ETH 或 USDC），不再赚手续费
   •   需要更多管理：可能得频繁调整区间（Gas 成本↑）
   •   更接近“主动交易”策略，不是纯被动

5️⃣ 你可以把它理解成
   •   v2：像在整个太平洋撒网，鱼很多地方都游不到你的网
   •   v3：在鱼群密集的海湾精准撒网，网更小但捕获率更高

# 2025-08-13

一、中文排版规范：让内容的“脸”更专业

在中文内容呈现中，排版是第一道门面关。
	•	字距与行距：保持字距均匀、行距适中（常用 1.4~1.75 倍行距），既避免密不透气的压迫感，也防止稀疏导致阅读断节。
	•	标题层级：从一级标题到三级标题递进清晰，字体大小、颜色或粗细应有区分，方便快速扫读。
	•	段落控制：单段落不宜过长，3~5 行为佳；适度分段能减少阅读疲劳。
	•	字体选择：正文优先选择易读性高的常规字体，艺术字体可作为点缀，用于封面、重点信息或广告位，切忌大面积使用。
	•	排版方向：横排适用于绝大多数数字内容，竖排可用于古风、书法等特殊主题。
	•	标点符号：注意中英文标点混用的规范，例如中文全角标点与汉字保持一致的字距，英文符号在中文语境中注意空格规则。

这样不仅提升了内容的专业性与可读性，也在潜移默化中塑造了品牌的品质感。

二、Twitter（X）运营：精准定位与高效触达

想在 Twitter 上建立影响力，先要锁定受众。
	•	定位与用户画像：通过数据工具分析粉丝的地域、兴趣、活跃时间，细化到年龄段与消费偏好，形成“画像标签库”。
	•	内容策略：在主题稳定性（主打领域）与热点灵活性（趋势话题）间找到平衡。可采用 70% 专业内容 + 20% 热点借势 + 10% 个人化分享的比例。
	•	发布时间：选择目标受众最活跃的时间段发布（可用 Twitter Analytics 数据验证），提升初始互动率。
	•	话题标签：结合热门标签与自定义品牌标签，前者拉新，后者沉淀社群。
	•	互动运营：及时回复评论与私信，定期举办 Q&A、投票、赠礼活动，让粉丝感受到“在被关注”。


三、社区搭建：从“有人”到“有人在说话”

一个成功的社区要从一开始就设计好目标与规则。
	•	核心目标：明确社区的使命（例如：行业交流、技能提升、兴趣陪伴）和关键成果指标（如活跃率、留存率、转化率）。
	•	受众需求：在建立前调研成员痛点与期望，确保社区内容有回应、有价值。
	•	平台选择：
	•	Discord：适合多频道管理与实时语音沟通，适配跨时区用户。
	•	Slack：适合以项目协作、任务管理为核心的团队型社区。
	•	Telegram：轻量级、易传播，适合快速聚拢人群。
	•	社区规则：明确发言规范、禁言条件、内容分类，降低后期管理成本。

四、工具与流程：用 Notion 让节奏可视化

在内容与活动管理中，Notion 是一款灵活且高度可定制的中枢工具。
	•	排期管理：按时间线（Timeline）、主题（Tags）或渠道（Platform）分类，方便不同维度查看进度。
	•	提醒功能：为重要节点（如活动启动、预热、上线、复盘）设置提醒，防止遗漏。
	•	资源库：集中管理素材、话题库、参考链接，形成可复用资产。

五、活动运营 SOP：让每次活动可复制、可优化
	•	策划阶段：明确活动目标（拉新、促活、转化）、受众画像、预期数据指标。
	•	执行阶段：设定执行清单，明确每个环节的责任人、完成时间、交付物。
	•	复盘阶段：收集数据与成员反馈，分析亮点与问题，形成优化建议。
	•	标准化沉淀：将最佳实践整理为 SOP 文档或模板，下次活动可直接套用，减少启动成本。

# 2025-08-11

核心流程概览与节点说明

一、核心流程概览

流程以 execute() 方法为起点，围绕 订单交易执行 展开，主要步骤如下：
	1.	参数校验与签名验证
	•	区分 Single / Bulk 订单类型
	•	通过哈希计算与签名解析，验证交易合法性
	2.	订单匹配与状态处理
	•	校验订单是否可匹配
	•	根据业务逻辑标记订单为 取消 或 完成 状态
	3.	资金与手续费流转
	•	处理 ETH / Token 支付
	•	拆分 平台手续费 与 订单手续费
	4.	NFT 资产交互
	•	兼容 ERC721 / ERC1155 标准
	•	最终触发链上事件 emit EVENT

二、关键节点说明

1. 签名验证分支（Single vs Bulk）
	•	Single 订单
	•	生成 orderHash
	•	通过 _hashToSign(orderHash) 计算签名哈希
	•	直接验证签名有效性
	•	Bulk 订单
	•	从签名数据解析 Merkle 树根哈希
	•	通过 hashToSign(root) 计算最终哈希
	•	适配批量交易的 多签 / Merkle 树验证逻辑


2. 资金转移核心逻辑
	•	ETH 余额校验
	•	通过 balanceETH = price 确保买方支付能力
	•	覆盖最大平台费扣除场景
	•	手续费拆分
	•	平台手续费 → _transferFees() 转账至 feesAddr
	•	订单手续费 → 买卖双方撮合扣费
	•	Token 转账适配
	•	通过 _transferTo() 处理支付
	•	支持 price - sellerFee 扣减逻辑
	•	兼容不同 NFT 资产类型的 Token 转账规则

三、适用场景与扩展
	•	智能合约开发
	•	对齐 Solidity 代码逻辑
	•	可直接关联 validateSignatures、executeFundTransfer 等方法
	•	便于合约调试与测试
	•	业务流程文档
	•	面向产品 / 运营解释链上交易闭环
	•	可补充 “手续费计算规则”“NFT 铸造条件” 等业务信息
	•	审计与合规
	•	标记 revert 回退节点
	•	清晰呈现资产转移路径
	•	便于排查交易失败原因与资金安全风险


四、文件使用建议
	•	编辑器兼容
	•	推荐使用支持 Mermaid 的工具（如 VSCode + Mermaid 插件、Obsidian）直接渲染流程图
	•	逻辑对齐代码
	•	如需与智能合约完全映射，可补充方法参数、事件参数（例如 emit EVENT 携带的订单哈希、资产类型）
	•	扩展业务规则
	•	在关键节点（如 “判断支付 Token 类型”）添加注释
	•	说明支持的 Token 列表、手续费比例等

# 2025-08-10

Web3法律知识分享会回放学习：
1. 常见罪名
	•	集资诈骗（ICO）
	•	非法吸收公众存款（ICO）
	•	组织、领导传销活动（与项目本身无关，通常与拉新分销方式有关）
	•	开设赌场
	•	非法经营（尤其是涉及结算业务）


2. 国内合规红线——ICO
	•	对发币类项目需高度谨慎
	•	必须具备海外合规牌照
	•	避免触碰中国大陆的金融监管红线


3. 资金汇集的风险要点
	•	无差别资金汇集风险极高
	•	涉及交易所或虚拟货币基金的场景
	•	是否直接或间接涉及法币交易
	•	海外合规性是否完备（包括 AML 反洗钱 与 KYC 用户身份验证）
	•	技术上是否有效屏蔽中国大陆用户

4. 传销与赌场风险
	•	回避多级返佣、团队奖励等传销式模式
	•	NFT 项目模式需特别审查其分销与返利机制


5. 难以判定的风险领域
	•	洗钱链条的一环，或外汇兑换媒介
	•	资金流向和来源路径对于非管理层人员往往难以彻查
	•	一旦出现不明资金流，应提高警惕并暂停参与


6. 从业者防御性判断清单
	•	项目是否在明确、受监管的司法辖区注册
	•	是否为取得合规资质而进行第三方审计/安全测试
	•	是否具备KYC、AML等反洗钱与用户身份识别制度
	•	是否公开披露项目负责人、团队背景、资金来源路径等关键信息

7. 法律盲区与“知情推断”

# 2025-08-08

智能合约基础与核心要点学习笔记

1. 什么是智能合约？

定义
智能合约是一段运行在区块链（主要是以太坊）上的代码，用于自动执行预先设定的逻辑（如转账、数据存储、条件触发）。它无需中介参与，代码即合约。

特性
	•	去中心化：部署在区块链节点网络中，没有单一控制方。
	•	不可篡改：一旦部署，代码无法直接修改。
	•	透明公开：代码逻辑对所有人可见。
	•	自动执行：符合条件时自动运行，无需人工干预。

我的理解
它就像一台 自动贩卖机：只要你投币（支付 ETH），就会自动给出饮料（执行合约逻辑），全程不需要售货员，而且机器的内部规则是公开透明、无法随意改的。


2. 核心知识点

(1) Solidity 语言
	•	以太坊智能合约的主要编程语言，语法类似 JavaScript，但更严格，因为代码漏洞可能直接导致资金损失。
	•	支持面向对象编程，能实现条件判断、循环、数据结构等复杂逻辑。
	•	例：一个 payable 函数可以接收 ETH 并记录发送方余额。

我的理解
Solidity 就像“区块链上的 JavaScript”，但多了资金安全的高压线——写错就可能烧钱。


(2) 运行环境与 Gas
	•	智能合约运行在 EVM（以太坊虚拟机） 中，这是一种去中心化的“沙盒”环境，确保执行安全且结果一致。
	•	执行智能合约需要支付 Gas 费（以 ETH 计价），费用取决于操作复杂度和当时的网络需求。
	•	简单转账 → Gas 低
	•	复杂循环 → Gas 高

我的理解
Gas 就像电费：用多少算多少，写高效的合约能帮你“省电”。


(3) 重要术语
	•	EVM：执行合约的虚拟机环境。
	•	ABI（应用程序二进制接口）：描述合约对外可调用的函数和参数的接口文件，类似 API。
	•	Gas Limit：单笔交易允许的最大 Gas，用来防止死循环堵塞网络。


3. 实操体验
	•	工具：Remix IDE（在线智能合约开发工具）。
	•	步骤：
	1.	连接 MetaMask 钱包。
	2.	编写简单的 ERC-20 代币合约。
	3.	部署到测试网（如 Sepolia）。
	4.	调用合约函数（如转账、查询余额）。

感受
部署过程需要手动设置 Gas 费，和链上钱包交互的感觉很真实。



4. 智能合约安全

常见漏洞
	1.	重入攻击（Reentrancy）
	•	攻击者在合约更新状态前反复调用提取资金的逻辑。
	•	案例：2016 年 DAO 黑客事件（损失 360 万 ETH）。
	•	防御：使用 Checks-Effects-Interactions 模式——先检查条件，再更新状态，最后调用外部合约。
	2.	整数溢出/下溢
	•	早期 Solidity 未自动检查数值边界，可能导致计算错误。
	•	防御：使用 SafeMath 库或 Solidity 0.8+ 内置检查。
	3.	未授权访问
	•	函数缺少访问控制，导致敏感操作被外部调用。
	•	防御：使用 require 或 onlyOwner 修饰符。

我的疑问
合约不可修改，那出现漏洞怎么办？
答案：通过 代理模式（Proxy Pattern） 分离逻辑与数据，实现合约升级——更新逻辑合约而保留原有数据。



5. Gas 优化

为什么要优化？
Gas 高 → 用户成本高 → 用户体验差 → 交易可能失败。

常见优化方法
	•	减少存储操作：storage 操作成本比 memory 高很多，能用内存就别用存储。
	•	避免复杂循环：用 mapping 替代数组遍历。
	•	用事件（Event）记录非核心数据：比存储便宜。
	•	用合适的数据类型：如 uint8 替代 uint256（在取值范围允许的情况下）。

我的理解
Gas 优化就像写高性能代码，既要正确，还要省油。


6. 常见开发框架

工具	特点	适用场景
Remix	在线 IDE，简单易用	新手入门、原型开发
Hardhat	插件丰富，本地调试强大	中大型项目
Truffle	生态成熟，集成 Ganache	初学者、本地开发
Foundry	Rust 编写，执行速度快	高性能开发、测试自动化

我的感受
	•	新手用 Remix 上手最快；
	•	想做专业项目 → Hardhat / Foundry；
	•	Truffle 虽老牌，但生态稳。


7. 以太坊生态与安全
	•	智能合约是以太坊的核心，支持 DApp、DeFi、NFT、DAO 等多种场景。
	•	以太坊 2022 年完成 PoW → PoS 转型（合并），降低能耗并提高 Gas 费可预测性，但合约优化仍必不可少。
	•	51% 攻击：在 PoS 下成本极高，需要大量质押 ETH 才可能发动。


8. 与传统编程的区别

智能合约 vs Python/JavaScript
	•	更强调 安全性（不可篡改、资金安全）。
	•	要考虑 Gas 成本（经济激励与性能优化结合）。
	•	运行在去中心化网络，不是本地机。
	•	部署与调试门槛更高，但逻辑上仍类似面向对象编程。
